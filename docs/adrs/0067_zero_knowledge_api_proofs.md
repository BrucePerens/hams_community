# ADR 0067: Zero-Knowledge API Proofs (HMAC-SHA256)

## Status
Accepted

## Context
Previously, real-time streaming endpoints (like the DX Cluster WebSocket Firehose) required the user to transmit their raw `adif_api_secret` in the URL query string to authenticate. Because query strings are inherently logged in plaintext by Nginx, Cloudflare, and browser histories, this created a severe credential leakage vulnerability.

## Decision
We strictly mandate the use of **Proof-of-Possession** (Zero-Knowledge) cryptographic handshakes for all authenticated platform connections where standard session cookies are unavailable.

Users and headless clients MUST NEVER transmit their raw API secret over the network. Instead, they must transmit:
1. Their `user` identifier (Callsign).
2. A Unix `timestamp` representing the current exact time.
3. A highly entropic `nonce` (Number Used Once) string.
4. An HMAC-SHA256 `signature` generated by hashing the `timestamp|nonce|payload` using their private API secret.

The server recalculates the expected signature. If it matches, the server verifies the Nonce against an ephemeral Redis cache. If the Nonce has been seen within the 5-minute TTL window, the connection is violently dropped as a replay attack.

### Exemptions for External Protocols
This mandate explicitly **does not apply** to external APIs or standardized legacy protocols where the client implementation is not in our control. For example, standard DDNS clients (hardware routers) and ACME webhooks (Certbot) cannot perform custom HMAC-SHA256 signing. For these specific integration points, transmitting microscopically scoped tokens directly is an accepted interoperability risk.

When utilizing these exemptions, the platform **MUST** enforce transport encryption (TLS/HTTPS) to protect the credentials in transit if at all possible. However, it must be explicitly understood that transport encryption provides **zero protection** against secret divulging or replay attacks if an attacker intercepts the data from the client's side of the connection, or via endpoint logging mechanisms (e.g., compromised local routers, browser history, or reverse proxy `access.log` files).

## Consequences
* Credentials are no longer leaked in reverse proxy logs.
* Replay attacks are entirely neutralized. The vulnerability window is reduced from 5 minutes to exactly 0 seconds.
* Integrators must implement local HMAC hashing in their desktop loggers or custom Javascript before initiating WebSocket connections.
